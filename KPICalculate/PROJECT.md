# 通用绩效分析 CLI 项目说明

## 项目目标

本项目用于分析虚拟或真实项目团队的周报、会议记录等文本材料，适用于创意设计、运营活动、咨询项目、软件开发等不同类型项目。

第一版只做命令行程序，不做 Web 页面，不做数据库。核心目标是跑通：

```text
输入项目材料 -> 大模型批量完成两两比较 -> 系统计分排序 -> 输出报告和日志
```

系统不能硬套软件开发指标。比较时不能只看谁做得多，而要结合岗位职责、项目目标和本周材料，判断成员在其岗位职责范围内对项目目标产生的相对价值。

成员画像、沟通风格和工作习惯只能作为岗位背景参考，不能作为预设扣分依据；比较必须以本周周报、会议发言和项目目标贡献为主要判断依据。

## 技术边界

- 运行方式：Python CLI
- 不使用数据库
- 不做 Web 页面
- 不使用 config 目录
- 大模型接口：OpenAI 兼容第三方 API
- API 地址、Key、模型名从 `.env` 读取
- 程序默认请求：`{LLM_BASE_URL}/chat/completions`
- 如果 `.env` 缺少必要变量，程序直接终止
- 第一版不使用 LangChain 或 LangGraph
- 并发使用 Python `asyncio`
- 可复用提示词单独放在 `prompts/` 目录，便于审查

`.env` 必需变量：

```env
LLM_BASE_URL=https://api.example.com/v1
LLM_API_KEY=your-api-key
LLM_MODEL=your-model-name
```

## 固定运行参数

除分析模式和模式 2 的维度选择外，第一版不做其他命令行交互。

固定参数：

- 输入目录：`input`
- 输出目录：`output`
- 并发数：`3`
- API 失败重试：`3`
- 请求超时：`120` 秒
- temperature：`0`
- 失败策略：跳过该场比较，记录异常，不计入总分
- 输出报告：Markdown + HTML
- 每次大模型交互必须保存完整日志
- 大模型调用按周、评价官、批次合并
- 每次大模型调用最多处理 `10` 组 pair

每次运行输出目录：

```text
output/
  latest_run.txt
  runs/
    run_YYYYMMDD_HHMMSS/
      reports/
        report.md
        report.html
      logs/
        run.log
        comparisons.jsonl
        llm_interactions.jsonl
        errors.jsonl
      artifacts/
        selected_dimensions.json
        run_snapshot.json
```

## 启动交互

程序启动后只询问分析模式：

```text
请选择分析模式：
1. 直接比较
2. 绩效维度比较
3. 评价官比较
4. 绩效维度 + 评价官比较
```

如果选择模式 2：

1. 系统调用大模型，根据项目背景、岗位背景、成员职位信息生成 5 到 10 个绩效维度。
2. 用户至少选择 3 个维度，并输入排序后的编号。

示例：

```text
请输入选择和排序后的维度编号：3,1,5,2
```

用户不需要填写权重。排序越靠前，提示词中优先级越高，但系统不做数值权重计算。

如果选择模式 4：

1. 系统调用大模型，根据项目背景、岗位背景、成员职位信息，自动选择 5 个更适合体现岗位侧重的绩效维度。
2. 用户不参与维度选择。
3. 自动选择的维度需要保存到输出目录，供审查。

## 分析模式

### 1. 直接比较

把项目组内每个人的每周工作材料进行两两比较，让大模型直接判断谁本周贡献更高。

模型只返回结构化胜负结果和简短原因，不需要长篇解释。

实现上，系统按周把多个 pair 合并到一次大模型请求中，而不是每一场比较单独请求。

### 2. 绩效维度比较

系统先生成 5 到 10 个适合当前项目的绩效维度。

用户至少选择 3 个维度，并对维度排序。系统基于这些维度进行两两比较。

### 3. 评价官比较

增加两个虚拟评价官：

- HRBP
- 业务主管

HRBP 偏关注：

- 岗位匹配
- 协作
- 稳定性
- 成长性
- 团队影响

业务主管偏关注：

- 业务结果
- 交付质量
- 项目推进
- 问题解决
- 目标贡献

两个评价官分别进行两两比较，系统再汇总结果。

### 4. 绩效维度 + 评价官比较

系统先让大模型自动选择 5 个更适合体现岗位侧重的绩效维度，再让 HRBP 和业务主管分别基于这些维度进行两两比较。

模式 4 不进行用户维度选择。大模型选择维度时必须结合：

- 项目目标
- 部门或岗位背景
- 成员职位与职责差异
- 不同行业项目的通用价值贡献

维度应避免只偏向某一类岗位，也不能硬套软件开发指标。

最终输出：

- HRBP 排序
- 业务主管排序
- 综合排序

综合排序规则：HRBP 积分 + 业务主管积分。两个评价官等权。

## 输入数据要求

标准输入结构后续再最终确认。当前实现需要能够表达以下信息：

- 项目背景
- 结构化项目范围，若有
- 部门或岗位背景
- 项目成员及职位信息
- 每周工作周报
- 每周会议记录
- 任务清单与最终状态，若有

如果项目背景缺失：

- 只能执行模式 1：直接比较
- 如果用户选择模式 2、3、4，系统应提示无法执行并降级或要求重新选择模式 1

如果成员职位信息缺失：

- 只能执行模式 1：直接比较
- direct 仍可运行，但提示“缺少职位信息，评价只能基于材料文本”

## 会议材料参与规则

每周比较时，提示词中加入本周会议记录全文。

但模型必须只采纳与 A/B 两人有关的内容。

成员是否有会议材料的确定性判断规则：

- 只看会议记录 `meetings[].speeches[]` 中是否有该成员发言
- 如果有该成员发言，视为该成员本周有会议材料
- 如果没有该成员发言，则不把会议材料视为该成员个人材料
- `participants` 只作为会议参与信息，不用于判定个人材料

## 缺材料规则

每周所有配置成员都参与两两比较。

对任意一场 A/B 比较：

- A 没有周报，且会议 `speeches` 中没有 A 的发言：A 视为缺材料
- B 没有周报，且会议 `speeches` 中没有 B 的发言：B 视为缺材料
- 只有 A 缺材料：B 自动胜，不调用大模型
- 只有 B 缺材料：A 自动胜，不调用大模型
- A/B 都缺材料：平局，不调用大模型
- A/B 至少都有一种材料：调用大模型比较

## 计分和排序规则

每周对所有成员两两比较。

单场计分：

- 胜：`1`
- 平：`0.5`
- 负：`0`
- 异常：不计入总分

先生成每周排名，再汇总生成整个项目周期总排名。

同分并列，不强行打破。

评价官模式下，每个评价官单独计分和排序；综合排序直接累加各评价官积分。

## 大模型批量比较规则

系统保留每一场两两比较结果，但大模型交互不按单场逐个请求。

默认合并粒度：

```text
周次 + 评价视角 + 批次
```

其中：

- 模式 1：每周按批比较所有 pair
- 模式 2：每周按批基于用户选择的维度比较所有 pair
- 模式 3：HRBP 每周按批比较，业务主管每周按批比较
- 模式 4：先自动选择 5 个岗位侧重维度，再由 HRBP 和业务主管分别每周按批比较

固定批大小：

```text
max_pairs_per_llm_call = 10
```

处理流程：

1. 每周生成所有成员 pair。
2. 先应用缺材料规则，自动判负或平局的 pair 不发送给大模型。
3. 需要模型判断的 pair 按 `max_pairs_per_llm_call` 分批。
4. 每个批次调用一次大模型。
5. 大模型必须返回该批次内所有 pair 的结构化比较结果。
6. 系统校验返回结果是否覆盖所有 pair。
7. 缺失、重复、成员不匹配、格式错误的 pair 记为异常，不计分。

批次调用失败处理：

- 如果某个批次 API 调用失败，重试整个批次
- 最多重试 `3` 次
- 如果 3 次后仍失败，批次内所有 pair 记异常，不计分

批次部分返回异常处理：

- 如果一个批次内部分 pair 返回有效，部分 pair 缺失或格式错误
- 有效 pair 正常计分
- 异常 pair 不做单独补救
- 异常 pair 记录到 `errors.jsonl`，不计分

这样可以降低耗时和成本，同时不改变“两两比较 + 系统汇总排序”的核心方法。

## 大模型返回格式

批量两两比较要求模型只返回 JSON：

```json
{
  "comparisons": [
    {
      "pair_id": "2025-W18__M001__M002",
      "a": "M001",
      "b": "M002",
      "winner": "A",
      "confidence": "medium",
      "reason": "A本周推进关键风险闭环更直接"
    }
  ]
}
```

允许值：

- `winner`: `A`、`B`、`TIE`
- `confidence`: `low`、`medium`、`high`

`confidence` 只用于审查，不参与计分。

`reason` 是简短原因说明，应控制在 50 个汉字以内，用于报告展示和审查，不参与计分。

如果模型返回格式错误：

- 尝试修复或重新请求
- 仍失败则记录异常
- 该场比较跳过，不计入总分

## 日志要求

每一次大模型交互都必须保存到 `llm_interactions.jsonl`。

至少包含：

- 时间
- 交互类型
- prompt 模板名
- 渲染后的完整 prompt
- 请求参数
- 原始响应
- 解析结果
- 批次内 pair 列表
- 耗时
- 错误信息

每一场比较都必须保存到 `comparisons.jsonl`。

至少包含：

- 周次
- 分析模式
- 评价官，若有
- 维度，若有
- 成员 A
- 成员 B
- winner
- score_a
- score_b
- reason
- 是否调用大模型
- 是否异常
- 异常原因，若有

异常单独保存到 `errors.jsonl`。

## 报告要求

生成 Markdown 和静态自包含 HTML。

报告至少包含：

- 运行概要
- 分析模式
- 项目背景
- 结构化项目范围
- 部门/岗位背景
- 成员信息
- 任务清单与最终状态
- 每周工作周报
- 每周会议发言
- 总排名
- 每周排名
- 评价官排名，若有
- 已选绩效维度，若有
- 自动选择的岗位侧重维度，若有
- 两两比较明细
- 每场比较的简短原因
- 异常比较数量和简要清单
- 日志文件路径

报告中只展示 `reason` 简短原因，不展示模型额外生成的长篇解释。即使模型原始响应包含额外解释，也只保存在日志中。
